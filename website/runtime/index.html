Runtime API

Direct server mode and Agent Kernel mode, mapped clearly.

The same operational core is available through a direct HTTP server on `127.0.0.1:8765` or an Agent Kernel runtime exposing core routes under `/api/v1/*` and custom operator routes under `/custom/*`.

Two runtime shapes

Direct HTTP mode is the simplest local path. Agent Kernel mode is the preferred long-running API runtime with both Agent Kernel endpoints and custom operator endpoints.

GET/health

Approval state, resolved config values, tool availability, and channels.

POST/plan

Plans a workflow without executing commands.

POST/run

Executes a workflow and enforces the mutation approval gate.

POST/chat

Synchronous JSON chat response for operator questions.

Direct mode
python3 -m microcloud_agent serve \
  --host 127.0.0.1 \
  --port 8765
Agent Kernel mode
AK_API__HOST=127.0.0.1 \
AK_API__PORT=8010 \
python3 -m microcloud_agent.agentkernel_app

Shared payload shape

Planning and execution use the same workflow context, which means operators and automation systems can switch from `plan` to `run` without changing the request structure.

Plan or run payload
{
  "workflow": "assess_health",
  "context": {
    "environment": "lab",
    "inventory": "ansible/inventories/lab/hosts.ini",
    "terraform_dir": "terraform/environments/lab",
    "service": "",
    "host": ""
  }
}
Chat payload

`{ "message": "what workflows do you support?" }`

Stream path

`POST /chat/stream` returns `text/event-stream` deltas.

Federation paths

`/openid-configuration`, `/oauth2/token`, and `/openapi/request` are part of the current HTTP surface.

Deterministic contract

The runtime is strongest where the contract is explicit.

Plan before mutation

Operators can inspect workflow steps before running them.

Run with approval

Mutating workflows fail unless `MICROCLOUD_AGENT_APPROVAL=approved` is set.

Return structure

Responses stay JSON-first instead of hiding behavior behind opaque chat copy.
