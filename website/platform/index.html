Platform

Specific operator features, not generic cloud claims.

The repository already has a clear feature surface. This page turns that into a product narrative without diluting it: workflows, tool adapters, remote execution, notifications, and identity-aware API federation.

Core runtime surface

The runtime supports local CLI execution, a direct HTTP server, an Agent Kernel bridge, a hardened Docker image, and host-operator deployment over SSH or Tailscale SSH.

  • Deterministic workflow planning before execution
  • Runtime discovery of actual binary availability on the host
  • Approval-gated mutating workflows
  • Structured health, plan, run, and chat outputs
Runtime modes

CLI, direct HTTP, Agent Kernel, Docker, systemd

Execution paths

Local binaries or remote wrappers via `ssh` and `tailscale ssh`

Notification

Mattermost outbound summaries and raw messages

Federation

OIDC discovery, OAuth2 client credentials, and OpenAPI request channel

Workflow inventory

Real workflows already implemented in the repo.

assess_health

Inventory, fact gathering, `microcloud status`, `lxc list --format json`, and `terraform validate`.

bootstrap_cluster

Preflight, `terraform plan`, and `microcloud init` for cluster bring-up. Mutating.

upgrade_cluster

Upgrade-readiness flow across inventory, cluster status, and Terraform planning.

assess_operator_tooling

Checks `gh`, `code`, `ssh`, `docker`, `snap`, VPN, DNS, `dig`, reverse proxy, `playwright`, and `canvas`.

configure_single_node

Preflight plus `microcloud init` and status validation for single-node deployments. Mutating.

configure_multi_node

Uses inventory, `microcloud status`, `microcloud add <host>`, and `lxc list --format json`. Mutating and host-aware.

Tool adapters and remote boundaries

The agent resolves binaries from environment variables and reports availability through health responses. The important design choice is that Docker hosts the API runtime, but the real infrastructure binaries remain on the target system.
